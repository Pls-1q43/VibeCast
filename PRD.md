下面这份 PRD 已将核心架构改为：**Android 网页只承载文本输入，语音识别完全由 Android 默认输入法完成；Mac 端只负责文本镜像、目标 App 聚焦和发送。**

需要提前明确一个系统限制：网页聚焦文本框后可以唤起 Android 输入法，但通常不能直接命令微信输入法自动进入“语音输入”状态。用户仍可能需要点击输入法中的语音按钮。其余流程可以完全自动化。

# Android 远程语音输入与 macOS 多应用文本同步系统

## 产品需求文档 PRD

**文档版本：** V1.0
**产品代号：** VibeCast
**目标平台：** Android 手机浏览器/PWA + macOS
**目标用户：** 单用户、本地设备、局域网环境
**首批目标应用：** Codex、WorkBuddy、Notion、CodeBuddy
**文档用途：** 指导开发 Agent 完成产品设计、编码、测试、部署和交付

---

# 1. 产品概述

本产品用于将 Android 手机变成 macOS 的远程语音文本输入面板。

用户在 Android 手机上打开一个由 Mac 提供的局域网页。网页中展示 Codex、WorkBuddy、Notion、CodeBuddy 四个目标应用的输入区域。

用户点击某个应用对应的文本框后：

1. 网页选择该应用作为当前目标；
2. Mac 自动激活对应应用；
3. Mac 自动聚焦对应应用的聊天或文本输入框；
4. Android 唤起系统默认输入法；
5. 用户使用微信输入法或其他 Android 输入法的语音输入功能；
6. 语音识别结果进入手机网页的文本框；
7. 手机文本框中的内容实时同步到 Mac 对应应用的输入框；
8. 用户可以继续在手机端修改、删除、补充文字；
9. 所有修改继续同步到 Mac；
10. 用户点击手机网页上的“发送”按钮；
11. Mac 确认最终文本已同步后，执行目标应用对应的发送操作。

本产品不在 Mac 上进行语音识别，也不接收或传输音频。

---

# 2. 产品目标

## 2.1 核心目标

实现以下完整体验：

> 用户拿起 Android 手机，点击 Codex、WorkBuddy、Notion 或 CodeBuddy 对应的输入框，通过微信输入法说话。识别出的文字实时出现在 Mac 对应应用的输入框中。用户在手机上修改文字后，Mac 端内容同步变化。点击手机上的“发送”后，消息在对应 Mac 应用中发出。

## 2.2 体验目标

产品应满足：

* 手机端无需安装原生 Android App；
* 手机端不负责语音识别；
* 用户可以继续使用自己熟悉的微信输入法；
* Mac 端不运行 Whisper 或其他语音模型；
* 手机端文本是当前输入会话的唯一数据源；
* Mac 端输入框是手机文本的实时镜像；
* 不通过逐字模拟键盘实现同步；
* 不允许文字误输入到错误的应用或错误的输入框；
* 点击发送前必须确认最终文本已经同步；
* 网络中断时不得误发半成品文本；
* 全部数据原则上只在局域网内传输。

## 2.3 非目标

第一版本不要求实现：

* Android 原生应用；
* 浏览器录音；
* 浏览器麦克风权限；
* Mac 端语音识别；
* 云端语音识别；
* 多用户协作；
* 公网远程控制；
* Mac 向手机反向同步用户在 Mac 上进行的文本编辑；
* 手机直接控制 Mac 鼠标；
* 通用远程桌面；
* 自动识别任意 App 的任意输入框；
* 强制 Android 输入法自动打开语音面板。

# 3. 核心设计原则

## 3.1 Android 输入法负责语音识别

网页只提供标准多行文本框。

用户点击文本框时，由 Android 系统唤起默认输入法。用户在微信输入法中使用语音输入，输入法负责把识别结果写入网页文本框。

系统不得：

* 请求网页麦克风权限；
* 上传音频；
* 保存音频；
* 调用 Whisper；
* 调用 Android SpeechRecognizer；
* 调用第三方语音识别 API。

## 3.2 手机端文本是唯一事实源

在一次输入会话中：

* 手机文本框内容为权威版本；
* Mac 端输入框只是镜像；
* 手机端每次编辑都以“完整文本快照”同步；
* Mac 端手动编辑可能被下一次手机同步覆盖；
* 用户在同步期间不应直接编辑 Mac 输入框。

不采用字符级增量模拟，不要求计算复杂文本差异。

## 3.3 发送必须是两阶段操作

点击“发送”后必须按以下顺序执行：

1. 将手机端最新完整文本发送到 Mac；
2. 等待 Mac 返回对应版本已经写入成功；
3. 再执行目标应用的提交动作；
4. Mac 返回发送成功或失败；
5. 手机端根据结果更新状态。

严禁收到发送请求后直接按回车，而不确认最终文本是否已经完成同步。

## 3.4 禁止向未知输入框注入文本

Mac 端必须确认：

* 当前目标应用正确；
* 当前输入控件属于该目标应用；
* 当前输入控件符合该应用的配置；
* 当前会话与目标输入控件绑定关系仍然有效。

如果无法确认，不得把文本粘贴到当前随机焦点位置。

# 4. 目标用户流程

## 4.1 首次配置流程

1. 用户在 Mac 上安装并启动服务。
2. 服务申请 macOS“辅助功能”权限。
3. Mac 显示服务状态、局域网地址和配对二维码。
4. 用户使用 Android 手机扫描二维码。
5. 手机网页保存设备配对令牌。
6. 用户在 Mac 配置四个目标：

   * Codex；
   * WorkBuddy；
   * Notion；
   * CodeBuddy。
7. 每个目标配置：

   * 应用名称；
   * macOS Bundle ID；
   * 激活方式；
   * 聚焦输入框方式；
   * 发送方式；
   * 是否发送后清空手机草稿。
8. 配置完成后进行一次输入和发送测试。

## 4.2 日常输入流程

1. 用户打开手机桌面的 VibeCast。
2. 页面连接 Mac 服务。
3. 页面展示四个应用卡片。
4. 用户点击 Codex 卡片中的文本框。
5. Mac 激活 Codex，并聚焦其聊天输入框。
6. Android 显示默认输入法。
7. 用户点击微信输入法的语音按钮。
8. 用户说话。
9. 识别文字进入手机文本框。
10. 网页将文本同步到 Mac。
11. 用户继续说话或修改文字。
12. Mac 输入框持续与手机文本保持一致。
13. 用户点击“发送”。
14. 服务确认最终文本同步。
15. Mac 执行 Codex 的发送操作。
16. 手机显示“已发送”。
17. 根据目标配置清空或保留草稿。

## 4.3 切换目标流程

1. 用户当前正在编辑 Codex 草稿。
2. 用户点击 WorkBuddy 文本框。
3. Codex 会话暂停。
4. Codex 草稿保留在手机本地。
5. Mac 激活 WorkBuddy。
6. Mac 聚焦 WorkBuddy 输入框。
7. WorkBuddy 草稿同步到其输入框。
8. 后续输入只允许写入 WorkBuddy。
9. Codex 和 WorkBuddy 草稿不得互相覆盖。

# 5. 手机端页面需求

## 5.1 页面整体布局

首页展示四个目标应用卡片：

* Codex；
* WorkBuddy；
* Notion；
* CodeBuddy。

每张卡片包含：

* 应用名称；
* 应用图标或首字母图标；
* 多行文本输入框；
* 当前同步状态；
* “发送”按钮；
* “清空”按钮；
* 可选的“重新聚焦”按钮。

推荐布局：

* 手机竖屏下单列排列；
* 每张卡片高度足以显示三至六行文字；
* 当前选中的应用卡片明显高亮；
* 发送按钮应足够大，便于单手操作；
* 页面底部显示 Mac 连接状态。

## 5.2 文本框

必须使用标准 `<textarea>`，不优先使用 `contenteditable`。

原因：

* Android 输入法兼容性更高；
* 微信输入法语音结果更容易正常写入；
* 更容易获取完整文本；
* 更容易获取光标和选区；
* 更容易处理组合输入事件；
* 页面刷新后更容易恢复草稿。

文本框要求：

* 支持多行；
* 支持中文；
* 支持中英文混合；
* 支持标点；
* 支持长文本；
* 支持粘贴；
* 支持撤销；
* 支持输入法候选；
* 支持语音输入；
* 支持手机端修改和删除；
* 点击后选择对应目标；
* 点击后向 Mac 发出“激活并聚焦目标”请求。

## 5.3 Android 输入法行为

用户点击文本框后，浏览器应通过标准聚焦行为唤起 Android 默认输入法。

系统不能承诺直接打开微信输入法的语音模式。

页面可在首次使用时显示提示：

> 点击文本框后，在微信输入法中点击语音按钮开始说话。

不得在网页中模拟一个假的录音按钮，避免让用户误以为网页正在录音。

## 5.4 独立草稿

四个目标必须各自维护独立草稿。

例如：

```text
Codex 草稿：检查这个项目中的异常处理。
WorkBuddy 草稿：帮我总结今天的会议内容。
Notion 草稿：记录一个关于远程输入的产品想法。
CodeBuddy 草稿：重构当前函数并增加单元测试。
```

切换目标时：

* 不清空原目标草稿；
* 新目标显示自己的历史草稿；
* 不把原目标文本同步到新目标；
* 页面刷新后应恢复未发送草稿。

草稿默认保存在手机浏览器本地存储中。

不得将草稿上传到第三方服务器。

## 5.5 同步状态

每个应用卡片应显示以下状态之一：

* 未连接；
* 待选择；
* 正在聚焦；
* 已聚焦；
* 正在同步；
* 已同步；
* 输入法编辑中；
* 目标失焦；
* Mac 应用未运行；
* 同步失败；
* 等待重连；
* 正在发送；
* 已发送；
* 发送失败。

状态必须用文字表示，不能只依赖颜色。

## 5.6 发送按钮

发送按钮满足：

* Mac 未连接时禁用；
* 目标未成功聚焦时禁用；
* 文本为空时默认禁用；
* 有未确认同步版本时显示“同步中”；
* 点击后立即锁定，避免重复点击；
* 收到发送成功结果后恢复；
* 发送失败时不得自动重试提交动作；
* 发送失败后允许用户手动重试；
* 发送成功后是否清空草稿由目标配置决定。

## 5.7 清空按钮

清空按钮行为：

1. 弹出轻量确认；
2. 清空手机端对应草稿；
3. 向 Mac 同步空字符串；
4. 不执行发送；
5. 保持当前目标和焦点。

## 5.8 连接状态

页面必须显示：

* 当前 Mac 名称；
* 是否已连接；
* 最近一次成功同步时间；
* 局域网连接异常提示；
* 是否需要重新配对。

断线期间：

* 用户仍可在手机端编辑草稿；
* 不得显示为已同步；
* 不得允许发送；
* 重连后自动将当前完整草稿同步到对应目标；
* 不得逐条重放断线期间所有历史编辑事件。

# 6. 输入法事件处理要求

网页需要正确监听：

* `focus`；
* `blur`；
* `beforeinput`；
* `input`；
* `compositionstart`；
* `compositionupdate`；
* `compositionend`；
* `selectionchange`；
* `keydown`，仅用于必要的快捷行为。

## 6.1 组合输入

中文输入法可能在文字正式提交前产生组合状态。

系统应记录：

```text
isComposing = true / false
```

默认策略：

* `compositionstart` 后进入输入法编辑状态；
* 组合期间允许发送文本预览快照；
* 组合期间的同步应进行节流；
* `compositionend` 后立即发送最终文本快照；
* 点击“发送”时如果仍处于组合状态，应等待组合结束；
* 不得把尚未提交的拼音字母误作为最终内容发送。

## 6.2 同步节流

为了兼顾实时性和稳定性：

* 普通输入事件建议使用 80～150 毫秒防抖；
* 组合输入期间建议使用 200～300 毫秒节流；
* `compositionend` 后立即同步；
* 点击发送前立即同步，不经过防抖等待；
* 只保留最新待发送文本快照；
* 旧快照不得覆盖新快照。

## 6.3 完整快照同步

每次同步发送完整文本，而不是只发送新增字符。

示例：

```json
{
  "type": "text_snapshot",
  "sessionId": "session-uuid",
  "targetId": "codex",
  "revision": 18,
  "text": "请检查当前项目中未处理的异步异常。",
  "selectionStart": 16,
  "selectionEnd": 16,
  "isComposing": false,
  "clientTimestamp": 1781760000000
}
```

Mac 只允许应用比当前版本号更高的快照。

# 7. Mac 端服务需求

## 7.1 服务职责

Mac 服务负责：

* 提供手机网页；
* 管理手机配对；
* 建立 WebSocket 长连接；
* 接收文本快照；
* 激活目标应用；
* 聚焦目标输入框；
* 将手机文本镜像到目标输入框；
* 返回同步确认；
* 执行发送动作；
* 返回发送结果；
* 提供目标应用配置；
* 提供诊断日志；
* 显示局域网访问地址；
* 开机自动启动。

Mac 服务不负责：

* 接收音频；
* 语音识别；
* 文本润色；
* 自动改写用户内容；
* 保存长期文本历史。

## 7.2 macOS 权限

首次启动时必须检测辅助功能权限。

若无权限：

* 明确显示权限缺失；
* 提供进入系统设置的按钮；
* 禁止执行文本注入；
* 禁止执行发送；
* 手机端显示“Mac 缺少辅助功能权限”。

不应申请无必要的权限，例如：

* 麦克风；
* 摄像头；
* 屏幕录制。

除非后续实现确实需要，否则不得要求屏幕录制权限。

## 7.3 服务形态

推荐提供一个 macOS 菜单栏应用。

菜单栏界面至少包含：

* 服务运行状态；
* 当前局域网地址；
* 手机连接数量；
* 当前目标；
* 辅助功能权限状态；
* 打开配置页面；
* 显示配对二维码；
* 重启服务；
* 查看日志；
* 退出程序。

## 7.4 开机启动

用户可配置：

* 登录 Mac 后自动启动；
* 默认开启局域网服务；
* 恢复上次目标配置；
* 不自动恢复未完成发送操作。

推荐配置结构：

```yaml
targets:

    activationMode: bundle_id
    focusMode: shortcut

    displayName: WorkBuddy

  notion:
    displayName: Notion
```

安装向导应支持从当前运行应用中选择目标程序，自动读取其 Bundle ID。

# 9. 应用激活与输入框聚焦

## 9.1 激活方式

优先通过 Bundle ID 激活应用。

支持：

* 激活已经运行的应用；
* 根据配置启动未运行的应用；
* 等待应用窗口出现；
* 获取应用前台状态；
* 检测启动超时；
* 返回应用不存在或无法启动错误。

## 9.2 聚焦策略

每个目标应用可选择以下聚焦策略。

### 策略一：应用快捷键

激活应用后执行用户配置的快捷键，用于聚焦聊天输入框。

这是优先推荐的方式。

### 策略二：macOS Accessibility

通过辅助功能 API 查找：

* 当前窗口；
* 可编辑文本控件；
* 文本区域；
* 搜索框；
* 输入框；
* 当前已聚焦控件。

需要为不同应用建立匹配规则。

### 策略三：恢复上次焦点

激活应用后，使用应用上一次保持的输入焦点。

此方式适合用户已经预先点击输入框的场景。

### 策略四：自定义自动化动作

允许用户配置：

* 多个连续快捷键；
* 等待时间；
* 菜单命令；
* AppleScript；
* Hammerspoon 动作；
* 自定义脚本。

### 不推荐策略：固定屏幕坐标

固定坐标只能作为明确标记的实验性降级方式，默认关闭。

窗口位置、缩放和显示器发生变化后，坐标方案容易误操作。

## 9.3 目标元素绑定

成功聚焦后，Mac 应尽可能记录：

* 目标应用 PID；
* Bundle ID；
* 目标窗口；
* 输入元素引用；
* 输入元素角色；
* 输入元素标识；
* 当前会话 ID。

后续同步时必须验证该绑定仍然有效。

如果输入元素已失效：

1. 尝试重新聚焦；
2. 重新验证目标；
3. 成功后继续同步；
4. 失败则暂停同步；
5. 手机端显示目标失焦。

不得把文字写到系统当前任意焦点位置。

# 10. Mac 文本写入策略

## 10.1 首选方式

首选通过 macOS Accessibility API 直接设置输入控件的值。

目标效果：

```text
Mac 输入框内容 = 手机文本框完整内容
```

优点：

* 不需要逐字输入；
* 支持任意位置修改；
* 支持删除；
* 支持替换；
* 不依赖键盘布局；
* 不容易产生字符丢失；
* 不容易受输入法状态影响。

## 10.2 降级方式

当目标输入控件不支持直接设置值时，使用：

1. 激活并聚焦目标输入框；
2. 记录系统剪贴板内容；
3. 全选目标输入框现有内容；
4. 将手机完整文本放入剪贴板；
5. 执行粘贴；
6. 验证结果；
7. 恢复用户原剪贴板内容。

要求：

* 不得永久覆盖用户剪贴板；
* 不得逐字模拟输入；
* 不得在未验证目标焦点时执行全选；
* 不得把整个页面或文档误全选；
* 对 Notion 等复杂编辑器必须进行专门测试。

## 10.3 不允许的方式

禁止以如下方式作为主要同步方案：

* 对每个字符发送键盘事件；
* 根据手机端差异发送大量退格键；
* 通过固定坐标点击输入框后盲目粘贴；
* 未验证目标应用就执行 Command+A；
* 未验证目标输入框就覆盖文本。

# 11. 同步协议

推荐使用 WebSocket。

## 11.1 连接初始化

手机连接后发送：

```json
{
  "type": "hello",
  "protocolVersion": 1,
  "clientId": "android-device-uuid",
  "deviceName": "Android Phone",
  "pairingToken": "secret-token"
}
```

Mac 返回：

```json
{
  "type": "hello_ack",
  "serverName": "Jeffrey's Mac",
  "protocolVersion": 1,
  "targets": [
    {
      "id": "codex",
      "displayName": "Codex",
      "available": true
    }
  ]
}
```

## 11.2 选择目标

手机发送：

```json
{
  "type": "select_target",
  "sessionId": "session-uuid",
  "targetId": "codex"
}
```

Mac 返回：

```json
{
  "type": "target_status",
  "sessionId": "session-uuid",
  "targetId": "codex",
  "status": "focused"
}
```

## 11.3 文本快照

手机发送：

```json
{
  "type": "text_snapshot",
  "sessionId": "session-uuid",
  "targetId": "codex",
  "revision": 23,
  "text": "帮我检查当前分支中的错误。",
  "selectionStart": 13,
  "selectionEnd": 13,
  "isComposing": false
}
```

Mac 成功写入后返回：

```json
{
  "type": "text_ack",
  "sessionId": "session-uuid",
  "targetId": "codex",
  "revision": 23,
  "applied": true
}
```

## 11.4 发送请求

手机必须携带最终版本号：

```json
{
  "type": "send",
  "sessionId": "session-uuid",
  "targetId": "codex",
  "revision": 23
}
```

Mac 的处理规则：

* 如果版本 23 尚未应用，先应用版本 23；
* 如果 Mac 只收到低于 23 的版本，拒绝发送；
* 重新确认目标输入框；
* 重新确认输入内容；
* 执行发送动作；
* 返回结果。

成功返回：

```json
{
  "type": "send_result",
  "sessionId": "session-uuid",
  "targetId": "codex",
  "revision": 23,
  "success": true
}
```

失败返回：

```json
{
  "type": "send_result",
  "sessionId": "session-uuid",
  "targetId": "codex",
  "revision": 23,
  "success": false,
  "errorCode": "TARGET_NOT_FOCUSED",
  "message": "无法确认 Codex 输入框焦点"
}
```

# 12. 版本与并发控制

## 12.1 Revision

每个目标草稿维护独立递增版本号。

例如：

```text
Codex revision: 18
WorkBuddy revision: 7
Notion revision: 2
CodeBuddy revision: 41
```

Mac 端只应用更高版本。

旧网络包即使延迟到达，也不得覆盖新文本。

## 12.2 单一活动会话

第一版本只允许一个活动手机控制一个目标输入框。

如果第二台设备尝试连接：

* 默认只允许查看服务状态；
* 不允许同时修改；
* 或要求用户在 Mac 上确认接管。

## 12.3 目标切换

切换目标后：

* 原目标会话停止发送新快照；
* 原目标草稿保留；
* 新目标创建或恢复独立会话；
* 任何迟到的原目标快照只能写回原目标；
* 不得根据“当前前台 App”重新解释旧消息。

# 13. 发送动作配置

每个目标可以定义不同发送方式。

支持：

* Enter；
* Command+Enter；
* Control+Enter；
* 自定义快捷键；
* Accessibility 点击发送按钮；
* 自定义脚本；
* 仅同步，不执行发送。

配置示例：

```yaml
sendAction:
  type: keyboard
  key: enter
  modifiers: []
```

或：

```yaml
sendAction:
  type: accessibility_button
  titleContains: "发送"
```

发送动作只能执行一次。

如果 Mac 未能判断是否已发送：

* 返回结果为未知；
* 不自动再次执行；
* 由用户检查后决定是否重试。

# 14. Notion 特殊要求

Notion 的普通页面编辑区与聊天输入框不同，存在误覆盖整段文档的风险。

因此 Notion Profile 必须明确指定以下模式之一：

## 14.1 Notion AI 对话模式

目标是 Notion AI 输入框。

可配置对应快捷键或 Accessibility 元素。

点击发送后执行 Notion AI 的提交动作。

## 14.2 当前文本块模式

目标是用户当前已经聚焦的单个 Notion 文本块。

此模式默认：

* 只同步文本；
* 不自动执行发送；
* 手机“发送”按钮改为“完成”；
* 不允许对整个页面执行全选替换。

## 14.3 固定收件箱模式

系统打开固定 Notion 页面，并将文字插入指定位置。

该模式属于增强功能，不作为最初 MVP 的强制要求。

# 15. 安全需求

## 15.1 网络范围

默认只监听：

* 本机；
* 当前局域网。

不得默认暴露到公网。

## 15.2 配对

首次连接必须经过配对。

推荐流程：

1. Mac 生成随机令牌；
2. QR 码中包含局域网地址和一次性配对码；
3. 手机完成配对；
4. 服务换发长期设备令牌；
5. 后续请求携带设备令牌；
6. 用户可以在 Mac 上撤销已配对设备。

## 15.3 请求验证

必须验证：

* 配对令牌；
* WebSocket 来源；
* 消息结构；
* 目标 ID；
* 会话 ID；
* Revision；
* 文本长度；
* 消息频率。

## 15.4 文本保护

默认：

* 不保存语音；
* 不存在音频；
* 不记录完整文本日志；
* 日志中最多记录文本长度和哈希；
* 草稿只保存在手机本地；
* Mac 内存中的文本在会话结束后释放。

## 15.5 PWA 与 HTTPS

核心网页输入和同步功能不依赖网页麦克风权限，因此可以在局域网 HTTP 下运行。

但如果要求：

* 安装为完整 PWA；
* 使用 Service Worker；
* 获得更稳定的主屏幕应用体验；

则应提供 HTTPS。

可选实现：

* 本地可信证书；
* Tailscale HTTPS；
* 用户自有域名和反向代理。

无论使用何种方式，控制接口不得无认证暴露到公网。

# 16. 错误处理

## 16.1 Mac 应用未运行

行为：

* 尝试按配置启动应用；
* 等待窗口出现；
* 超时后返回失败；
* 手机显示“应用未运行或启动失败”。

## 16.2 输入框无法聚焦

行为：

* 尝试重新执行聚焦动作；
* 最多重试一次；
* 不得盲目粘贴；
* 手机显示“无法定位输入框”；
* 提供“重新聚焦”按钮。

## 16.3 用户在 Mac 切换应用

默认行为：

* 检测目标失焦；
* 暂停自动同步；
* 不向新应用输入文字；
* 手机显示“目标已失焦”；
* 用户点击“重新聚焦”后恢复。

不得为了保持同步而持续抢夺 Mac 焦点，除非用户在设置中开启“强制保持目标前台”。

## 16.4 网络中断

行为：

* 保留手机草稿；
* 禁用发送；
* 标记未同步；
* 自动尝试重连；
* 重连后发送最新完整快照；
* 不重放中间历史；
* 不自动发送。

## 16.5 Mac 进入睡眠

行为：

* 手机显示离线；
* 保留草稿；
* Mac 唤醒并重连后重新选择目标；
* 不恢复未完成的发送动作。

## 16.6 重复点击发送

行为：

* 第一次点击后立即锁定按钮；
* 同一 session、target、revision 的发送请求具有幂等性；
* Mac 不得对同一发送 ID 执行两次提交。

# 17. 性能要求

在正常家庭或办公局域网下：

* 页面首次打开时间小于 2 秒；
* WebSocket 连接时间小于 1 秒；
* 目标应用激活通常小于 1 秒；
* 普通文本修改后 500 毫秒内出现在 Mac；
* `compositionend` 后 500 毫秒内完成最终同步；
* 点击发送后 1 秒内开始执行发送动作；
* 单个草稿至少支持 10,000 个字符；
* 连续输入 30 分钟不得明显内存泄漏；
* 每秒多次输入事件不得造成 Mac UI 卡死。

性能目标不是逐字符零延迟，而是稳定、准确和不误输入。

# 22. 推荐技术架构

本节为推荐实现，不限制开发 Agent 采用等价方案。

## 22.1 Web 前端

推荐：

* TypeScript；
* Vite；
* 原生 DOM 或轻量框架；
* WebSocket；
* IndexedDB 或 localStorage；
* 可选 PWA Manifest；
* 响应式移动端设计。

避免引入过重的前端框架。

## 22.2 Mac 服务

推荐两种方案。

### 方案 A：Swift 原生菜单栏应用

优点：

* 直接使用 Accessibility API；
* 权限管理清晰；
* 更容易获得稳定 App 激活能力；
* 更适合长期运行；
* 更容易实现开机启动。

### 方案 B：Web 服务加 Swift Bridge

组成：

* Python FastAPI 或 Node.js 提供网页和 WebSocket；
* Swift Helper 负责 macOS Accessibility；
* 两者通过本地 Unix Socket 或标准输入输出通信。

不建议完全依赖浏览器自动化。

Hammerspoon 可以用于原型验证，但正式版本优先使用 Swift Accessibility API。

## 22.3 数据流

```text
Android 默认输入法
        ↓
手机网页 textarea
        ↓
input / composition 事件
        ↓
WebSocket 文本快照
        ↓
Mac 本地服务
        ↓
目标校验与 Revision 校验
        ↓
macOS Accessibility
        ↓
Codex / WorkBuddy / Notion / CodeBuddy 输入框
```

发送数据流：

```text
手机点击发送
        ↓
立即发送最终快照
        ↓
等待 text_ack
        ↓
发送 send 命令
        ↓
Mac 重新验证目标和文本
        ↓
执行 Enter / Command+Enter / 自定义动作
        ↓
返回 send_result
```

# 24. 验收测试

## 24.1 基本输入

前置条件：

* Mac 和 Android 在同一局域网；
* Codex 已配置；
* 微信输入法为 Android 默认输入法。

步骤：

1. 手机打开网页；
2. 点击 Codex 文本框；
3. 使用微信输入法输入“测试远程输入”；
4. 观察 Mac。

验收结果：

* Codex 被激活；
* Codex 输入框被聚焦；
* Mac 输入框完整显示“测试远程输入”；
* 手机显示已同步。

## 24.2 语音输入

步骤：

1. 点击 CodeBuddy 文本框；
2. 在微信输入法中启动语音输入；
3. 说“请检查当前项目中的类型错误”；
4. 等待输入法完成识别。

验收结果：

* 手机文本框出现识别文字；
* Mac CodeBuddy 输入框出现相同文字；
* 系统未传输音频；
* Mac 未运行语音识别模型。

## 24.3 手机端修改

初始文本：

```text
请检查当前项目中的类型错误
```

用户在手机上修改为：

```text
请检查当前项目中的类型错误，并生成修复方案。
```

验收结果：

* Mac 输入框最终内容与手机完全相同；
* 不存在重复文本；
* 不存在残留旧文本；
* 不存在错误光标位置造成的插入。

## 24.4 删除文字

用户删除手机文本框中的一段内容。

验收结果：

* Mac 对应部分同步删除；
* Mac 不通过连续退格实现；
* 最终完整文本一致。

## 24.5 发送

步骤：

1. 手机输入一段文字；
2. 等待状态显示已同步；
3. 点击发送。

验收结果：

* Mac 最终文本与手机一致；
* 发送动作只执行一次；
* 手机显示已发送；
* 根据配置清空草稿。

## 24.6 快速输入后立即发送

步骤：

1. 用户完成语音输入；
2. 在输入法刚结束后立即点击发送。

验收结果：

* 系统先同步最新版本；
* 不发送旧版本；
* 不漏掉句尾文字；
* 不重复发送。

## 24.7 切换应用

步骤：

1. 在 Codex 输入草稿；
2. 切换到 WorkBuddy；
3. 在 WorkBuddy 输入另一段草稿；
4. 切回 Codex。

验收结果：

* 两个草稿独立；
* 不串写；
* Codex 恢复原草稿；
* WorkBuddy 保留自己的草稿。

## 24.8 Mac 焦点被切换

步骤：

1. 手机正在控制 Codex；
2. 用户在 Mac 手动切换到其他应用；
3. 手机继续输入。

验收结果：

* 文字不得进入其他应用；
* 手机显示目标失焦；
* 点击重新聚焦后恢复同步。

## 24.9 网络断开

步骤：

1. 手机断开 Wi-Fi；
2. 编辑草稿；
3. 点击发送；
4. 恢复 Wi-Fi。

验收结果：

* 断线期间发送按钮不可用；
* 草稿不丢失；
* 恢复后同步最新完整文本；
* 不自动发送。

## 24.10 剪贴板保护

步骤：

1. Mac 剪贴板中预先复制一段内容；
2. 执行一次手机同步；
3. 在 Mac 粘贴原剪贴板。

验收结果：

* 原剪贴板内容仍然可用；
* 系统未永久覆盖剪贴板。

# 27. 已知限制

1. 网页点击文本框只能保证唤起 Android 输入法，不能保证自动进入微信输入法的语音模式。
2. 不同版本微信输入法对组合输入事件的表现可能不同，需要真机测试。
3. Electron 应用或复杂编辑器对 macOS Accessibility 的支持可能不完整。
4. Notion 普通页面不适合直接执行全选替换，需要限定输入目标。
5. 目标应用更新后，快捷键或 Accessibility 结构可能变化。
6. 手机端是同步期间的唯一事实源，Mac 端手动修改可能被覆盖。
7. 手机浏览器进入后台后，WebSocket 可能被 Android 暂停。
8. 如果需要长期后台连接，未来可能仍需开发 Android 原生壳应用。
9. 第一版本主要针对用户主动打开手机页面并操作的前台场景。
10. 局域网地址可能变化，应通过二维码、mDNS 或固定主机名改善访问体验。

# 28. 最终产品一句话定义

VibeCast 是一个由 Mac 提供网页、由 Android 输入法完成语音识别、并把手机文本实时镜像到指定 macOS 应用输入框中的局域网远程输入系统。
